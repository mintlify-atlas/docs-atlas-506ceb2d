CONIITI 2026 is built as a **full-stack SaaS application** using modern web technologies. The platform follows a **client-server architecture** with a React frontend, Express backend, and Supabase for authentication and data storage.

### Performance Tracking

The platform includes a custom performance monitoring system:

```typescript
// frontend/vite-project/src/utils/tracker.ts
useEffect(() => {
  const navStart = performance.now();
  trackEvent('PAGE_VIEW', location.pathname);

  const timer = setTimeout(() => {
    const loadTime = Math.round(performance.now() - navStart);
    trackEvent('PAGE_LOAD_TIME', location.pathname, 'Carga de ruta', loadTime);
  }, 100);

  const observer = new PerformanceObserver((list) => {
    list.getEntries().forEach((entry: any) => {
      if (entry.entryType === 'resource') {
        const isImg = ['img', 'image'].includes(entry.initiatorType);
        if (isImg) {
          trackEvent('IMAGE_LOAD', location.pathname, entry.name, 
                     Math.round(entry.duration), entry.transferSize);
        }
      }
    });
  });

  observer.observe({ type: 'resource', buffered: true });
}, [location.pathname]);
```

## API Communication

### Centralized API Client

All API calls go through a centralized `apiFetch` function that handles authentication:

```typescript
const API_URL = import.meta.env.VITE_API_URL || 'http://localhost:3000/api';

export const apiFetch = async (endpoint: string, options: RequestInit = {}) => {
  // Get session token
  const { data: { session } } = await supabase.auth.getSession();

  const headers: Record<string, string> = {
    'Content-Type': 'application/json',
    ...(options.headers as Record<string, string>),
  };

  // Add auth token if available
  if (session?.access_token) {
    headers['Authorization'] = `Bearer ${session.access_token}`;
  }

  const response = await fetch(`${API_URL}${endpoint}`, {
    ...options,
    headers,
  });

  if (!response.ok) {
    const errorData = await response.json().catch(() => ({}));
    const errorMessage = errorData.error || errorData.message || `Error: ${response.status}`;
    throw new Error(errorMessage);
  }

  return response.json();
};
```

### Fallback Strategy

The platform implements a **resilient fallback strategy** to handle API failures gracefully:

```typescript
const fetchAllData = async () => {
  setIsLoading(true);
  let ponenciasData: any[] = [];
  let ponentesData: any[] = [];
  let usuariosData: any[] = [];

  try {
    // Try individual endpoints separately
    try { ponenciasData = await getPonencias(); } 
    catch (e) { console.warn("API Ponencias falló"); }
    
    try { ponentesData = await getPonentes(); } 
    catch (e) { console.warn("API Ponentes falló"); }
    
    try { usuariosData = await getUsuarios(); } 
    catch (e) { console.warn("API Usuarios falló"); }

    // Use fallback data if API fails
    const savedConfs = JSON.parse(localStorage.getItem("site_conferences") || '[]');
    const finalConfs = ponenciasData.length > 0 ? ponenciasData : 
                       (savedConfs.length > 0 ? savedConfs : initialConferences);
    setConferences(finalConfs);
  } catch (err) {
    console.error("Error crítico:", err);
    setConferences(initialConferences); // Ultimate fallback
  }
};
```

<Info>
This approach ensures the frontend remains functional even if the backend is temporarily unavailable.
</Info>

## Key Architectural Decisions

### Why LocalStorage + API?

The platform uses a **dual-layer persistence strategy**:

1. **LocalStorage** for immediate feedback and offline capability
2. **API + Database** for cross-device synchronization and data integrity

This ensures a responsive user experience even with slow network connections.

### Why TypeScript?

TypeScript provides:
- **Type safety** to catch errors at compile-time
- **Better IDE support** with autocomplete and refactoring
- **Self-documenting code** through type definitions
- **Easier refactoring** as the codebase grows

### Why Modular Backend?

The modular architecture enables:
- **Separation of concerns** (routes, controllers, services)
- **Easy testing** of individual modules
- **Scalability** by adding new modules without affecting existing ones
- **Team collaboration** with clear ownership boundaries

### Why Supabase?

Supabase was chosen for:
- **Rapid development** with built-in authentication
- **PostgreSQL** for relational data integrity
- **Row-level security** for fine-grained access control
- **Real-time subscriptions** for future enhancements
- **Free tier** suitable for university projects
